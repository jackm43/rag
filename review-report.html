Trust Zone Platform — Repository Review

Review + build pass on /Users/jackm/personal/rag · branch fable · 2026-06-12

Headline. The platform is architecturally sound — single STS issuer, short-lived DPoP-bound sessions, explicit delegation, references-not-values, audit-everywhere — and the implementation largely honors it. This review covered the auth gateway, both SDKs (TS + Go), the CLI, and the ragbot worker. Alongside the review I shipped two new applications (AI Gateway service, browser chat client), updated the design doc, and applied the safe fixes below. The most material real issue found is the unauthenticated-by-design DatabaseService.Query RPC (arbitrary SQL for any admin token); the headline false positive was a claimed nil-pointer panic in the Go SDK that is actually nil-guarded.

Open findings to decide

design riskDatabaseService.Query executes arbitrary SQLsrc/services.ts:110

Any token with the ragbot/DatabaseService.Query scope can run arbitrary SQL against D1 (parameters are bound, but the statement itself is caller-supplied — DROP, cross-table reads, writing rag_settings). This is intentional admin tooling for a single operator, so it’s a deliberate trade rather than a bug — but worth either narrowing to read-only/allowlisted statements or gating behind a dedicated higher scope. Left as-is pending your call.

recommendDPoP proofs aren’t single-use (no JTI replay cache)infra/sdk/ts/src/verify/dpop.ts

Proofs are bound to method/URI/time and to cnf.jkt, but a captured proof can be replayed within the 300s window. A real fix needs a short-TTL jti store (D1 or a Durable Object on the gateway). Noted in the design’s remaining-hardening list.

recommendAudit rows lack request contextinfra/gateway/src/registry.ts:383

audit(actor, action, detail) stores no IP, user-agent, or decision outcome, so denials and abuse patterns are hard to correlate. Extend the idp_audit_log schema + helper with client metadata and an allow/deny field.

recommendschema.sql vs migrations driftinfra/gateway/schema.sql vs migrations/

schema.sql already contains columns that the 0001/0002 migrations also ADD COLUMN. A fresh DB initialized from schema.sql then running migrations will conflict. Pick one source of truth (recommend: schema.sql as the full snapshot, migrations as forward-only deltas, and don’t replay old migrations on a schema.sql-seeded DB).

recommendDuplicate logger implementationssrc/logger.ts ≡ infra/sdk/ts/src/logger.ts

Identical files; the ragbot worker should import the SDK logger to avoid drift.
