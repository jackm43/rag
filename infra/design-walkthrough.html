When service B calls service C because user U asked it to, the platform's answer is explicit delegation: each hop exchanges the inbound token plus proof of the service's own identity for a new token that names both. The subject never changes; the actor chain only grows. Below: the same user request crossing three applications.

Structural guarantees

  • The chain is append-only: each exchange nests the previous act value; deepest entry = first hop.
  • Max depth 3 by default — deeper chains are denied with an audited reason (design smell).
  • The chain rides in the gateway-signed JWT, so signature verification already covers it — no extra encoding or hashing.
  • Subject scopes are irrelevant to chaining: the delegation grant is the policy, so a broad user token cannot be laundered into unapproved audiences.

What auditors can ask

  • "Everything service X did on behalf of users last week" → filter actor_chain contains app:X.
  • "Who could ever reach audience Y" → registry query over delegation grants, not a code review.
  • "Was this inference attributable?" → every AI Gateway call logs the resolved chain and model; traces carry identity annotations.

When not to chain

Background work a service does for itself (cron, queue consumers) uses its service credential as subject — no fabricated user. The audit distinction between "for a user" and "for itself" is deliberate and preserved (Module 7, Module 8).

Ten normative principles — every module must satisfy them, and every future addition is evaluated against them. Badges show implementation honesty: what is live, what is partially live, and what is designed but not yet in code (Module 15's hardening backlog).
